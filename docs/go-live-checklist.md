# 桐叶售电代理系统上线 checklist

> 适用版本：重构版 v1.0  
> 文档用途：指导系统上线前的准备、培训、切换、监控及回滚。

---

## 1. 上线前准备

### 1.1 环境检查

- [ ] 生产服务器资源配置已确认（CPU / 内存 / 磁盘 / 网络带宽）。
- [ ] 服务器操作系统及依赖补丁已更新至稳定版本。
- [ ] Docker 与 docker-compose 已安装，版本符合项目要求。
- [ ] PostgreSQL 数据库版本兼容（建议 15+）。
- [ ] Python 3.12 运行环境已就绪。
- [ ] Node.js 构建环境已就绪（建议 18 LTS+）。
- [ ] 后端依赖已按 `requirements.lock` / `pyproject.toml` 安装完成。
- [ ] 前端依赖已按 `package-lock.json` / `pnpm-lock.yaml` 安装完成。
- [ ] 生产网络端口已规划并放行（后端 API、前端、数据库、管理后台）。
- [ ] 日志目录、文件存储目录、备份目录已创建并持久化挂载。
- [ ] 容器资源限制（CPU / 内存）已配置。

### 1.2 数据准备

- [ ] 制定原系统数据导出方案（客户、合同、代理、电价、用电数据、历史利润）。
- [ ] 完成原系统数据清洗（去重、格式标准化、缺失值处理、异常值修正）。
- [ ] 编写并评审数据迁移脚本（全量迁移 + 增量迁移）。
- [ ] 在测试环境执行全量数据迁移，并记录耗时。
- [ ] 迁移后数据校验：
  - [ ] 各表记录数与原系统一致。
  - [ ] 关键字段（客户编码、合同编号、代理关系）无丢失。
  - [ ] 外键关联关系正确。
- [ ] 验证利润计算结果与原系统一致（抽样核对）。
- [ ] 验证代理费结算结果与原系统一致（抽样核对）。
- [ ] 准备数据回滚脚本与切换前全量备份方案。
- [ ] 制定数据修复流程，明确异常数据责任人。

### 1.3 部署检查

- [ ] 后端服务可通过 `docker-compose up -d` 正常启动。
- [ ] 后端健康检查接口（health check）返回正常。
- [ ] 数据库迁移脚本（Alembic / 其他）已在生产环境执行成功。
- [ ] 前端构建产物生成成功，无编译错误。
- [ ] 前端静态资源已部署到 CDN 或反向代理目录。
- [ ] 定时任务已配置完成并验证：
  - [ ] `daily-profit-calc`：每日利润计算。
  - [ ] `monthly-aggregation`：月度利润汇总。
  - [ ] `commission-settlement`：代理费结算生成。
  - [ ] `price-effective`：电价生效切换。
  - [ ] `contract-expiry-reminder`：合同到期提醒。
- [ ] 反向代理（Nginx / Traefik）配置完成并测试通过。
- [ ] HTTPS 证书已配置，域名解析正确。
- [ ] 后台管理地址已配置，并限制访问来源。
- [ ] 环境变量（`.env` / secrets）已正确注入，未硬编码敏感信息。

### 1.4 安全检查

- [ ] JWT 密钥已更换为生产环境专用强随机字符串。
- [ ] JWT 访问令牌与刷新令牌过期时间符合安全策略。
- [ ] PostgreSQL RLS（行级安全）策略已启用并通过验证。
- [ ] CORS 白名单仅包含生产域名，未配置通配符。
- [ ] HTTPS 强制跳转已开启，HSTS 配置正确。
- [ ] 敏感接口已启用速率限制 / 防暴力破解（登录、导出等）。
- [ ] 数据库账号权限最小化（应用账号仅授予必要权限）。
- [ ] 日志脱敏策略已配置（不记录密码、token、身份证号）。
- [ ] 容器镜像来源可信，无已知高危漏洞。
- [ ] 备份加密策略已确认。

## 2. 用户培训计划

### 2.1 培训对象

- [ ] **系统管理员**（1 ~ 2 人）：负责用户、角色、权限、定时任务、监控与备份。
- [ ] **财务人员**（1 ~ 2 人）：负责利润核对、代理费审批、结算付款。
- [ ] **业务员**（若干）：负责客户、询价报价、合同、用电数据录入。
- [ ] **代理商**（关键代理代表）：负责查看自身客户与代理费。

### 2.2 培训内容（按模块）

| 培训对象 | 模块 | 重点内容 |
| --- | --- | --- |
| 管理员 | 系统管理、安全 | 用户/角色/权限、RLS、日志查看、备份恢复、定时任务监控 |
| 业务员 | 客户管理、询价报价、用电数据 | 新增客户、价差变更、合同管理、创建询价、提交报价、导入 96 点数据 |
| 财务 | 利润管理、分润结算 | 日/月利润查看、统计导出、代理费审批、结算付款 |
| 代理商 | 代理管理（只读） | 代理树查看、代理费查询、客户分布、利润分成 |

### 2.3 培训方式与时间安排

- [ ] **第 1 天**：管理员培训（2 小时），覆盖环境、权限、监控、备份。
- [ ] **第 2 天**：业务员培训（3 小时），覆盖客户管理、询价报价、用电数据。
- [ ] **第 3 天**：财务人员培训（2 小时），覆盖利润管理、分润结算。
- [ ] **第 4 天**：代理商培训（1 小时），覆盖只读查询与代理费对账。
- [ ] **第 5 天**：综合答疑与模拟演练（2 小时），按真实业务场景跑通全流程。
- [ ] 培训过程录制视频，并上传至内部知识库。
- [ ] 编制并发放《桐叶售电代理系统操作手册》。
- [ ] 建立培训问题收集表，跟踪未解决问题。

## 3. 上线当日 checklist

### 3.1 系统最终验证

- [ ] 登录页可正常访问，登录后跳转首页无误。
- [ ] 左侧菜单所有模块可正常打开，无 404 / 500 报错。
- [ ] 关键业务流程跑通：
  - [ ] 新增客户 → 创建询价 → 提交报价 → 确认合作 → 签订合同。
  - [ ] 录入用电数据 → 执行利润计算 → 查看日/月利润。
  - [ ] 生成代理费结算单 → 审批 → 标记付款。
- [ ] 各角色权限控制验证（不同角色看到不同数据）。
- [ ] 导入任务（96 点数据）可正常提交并查看结果。
- [ ] 导出功能（Excel / PDF）正常。
- [ ] 定时任务手动触发成功，执行结果无异常。

### 3.2 数据切换 / 切换窗口

- [ ] 确认切换窗口时间（建议选择业务低峰期，如周末夜间）。
- [ ] 切换前对原系统执行最终全量备份。
- [ ] 冻结原系统写操作（暂停新合同、新客户、新询价录入）。
- [ ] 执行最终增量数据迁移。
- [ ] 验证迁移后数据一致性（记录数、关键业务抽样）。
- [ ] 验证利润与代理费计算结果正确。
- [ ] 关闭原系统对外服务。
- [ ] 开启新系统对外服务，更新域名解析或反向代理指向。
- [ ] 发布新系统访问地址。

### 3.3 监控与告警

- [ ] 应用监控已接入（CPU / 内存 / 磁盘 / 网络 / 容器状态）。
- [ ] 数据库监控已接入（连接池、慢查询、锁等待）。
- [ ] 定时任务执行结果监控已配置。
- [ ] 错误日志告警已配置（HTTP 500、未捕获异常、关键业务异常）。
- [ ] 业务指标告警已配置（利润计算失败、导入任务失败、合同到期提醒失败）。
- [ ] 告警通知渠道已验证（邮件 / 企业微信 / 钉钉）。

### 3.4 用户通知

- [ ] 发布系统切换公告（切换时间、新地址、账号信息、注意事项）。
- [ ] 向所有用户发送新系统登录地址与初始密码。
- [ ] 提供操作手册与培训视频链接。
- [ ] 设立上线支持群 / 热线电话，并安排值班人员。
- [ ] 告知用户切换期间数据冻结范围及恢复时间。

## 4. 上线后观察项

### 4.1 关键指标监控

- [ ] 日登录用户数与活跃用户数。
- [ ] 客户、合同、询价的新增量是否符合预期。
- [ ] 日利润计算成功率（目标 ≥ 99%）。
- [ ] 月度利润汇总成功率。
- [ ] 代理费结算生成成功率。
- [ ] 96 点数据导入成功率（目标 ≥ 95%）。
- [ ] 页面平均响应时间（目标 ≤ 2 秒）。
- [ ] 关键接口 P99 响应时间。
- [ ] 数据库连接池使用率。
- [ ] 错误日志数量及类型分布。
- [ ] 代理费结算及时率。
- [ ] 合同到期提醒送达率。

### 4.2 问题响应流程

- [ ] 设立问题收集通道（企业微信群 / 工单系统 / 热线电话）。
- [ ] 定义问题分级标准：
  - [ ] **P0**：系统整体不可用，所有用户无法操作。
  - [ ] **P1**：核心业务流程阻塞（登录、利润计算、结算）。
  - [ ] **P2**：一般功能异常，存在替代方案。
  - [ ] **P3**：优化建议或 UI 体验问题。
- [ ] 明确各级问题责任人及响应时限：
  | 级别 | 响应时间 | 解决目标 |
  | --- | --- | --- |
  | P0 | 5 分钟 | 30 分钟内恢复或启动回滚 |
  | P1 | 15 分钟 | 2 小时内恢复 |
  | P2 | 1 小时 | 24 小时内处理 |
  | P3 | 1 工作日 | 排期优化 |
- [ ] 上线前 3 天每日召开复盘会，汇总问题与处理进展。
- [ ] 重大问题立即启动回滚预案，并通知相关负责人。

## 5. 回滚预案

### 5.1 触发条件

- [ ] 系统出现 P0 级故障，30 分钟内无法定位或恢复。
- [ ] 核心业务流程（登录、利润计算、代理费结算）大面积不可用。
- [ ] 数据迁移出现严重错误，导致利润、代理费、客户数据无法修正。
- [ ] 发生安全事件（数据泄露、未授权访问、敏感信息泄露）。
- [ ] 业务方明确要求回滚。

### 5.2 回滚步骤

1. [ ] **决策**：由项目经理或技术负责人确认触发回滚，并通知业务方。
2. [ ] **冻结**：立即冻结新系统所有写操作，阻止用户继续录入数据。
3. [ ] **通知**：通过公告、群消息、电话等方式通知所有用户暂停使用新系统。
4. [ ] **关闭**：关闭新系统对外访问（停止容器或切换反向代理）。
5. [ ] **恢复**：启动原系统服务，恢复至切换前状态。
6. [ ] **验证**：验证原系统核心业务（登录、查询、录入、报表）正常。
7. [ ] **通知恢复**：通知用户恢复使用原系统，并说明原因。
8. [ ] **排查**：保留新系统数据库、日志、容器快照，用于后续问题排查。
9. [ ] **报告**：形成回滚报告，记录触发原因、处理过程、改进措施。

### 5.3 数据恢复

- [ ] 确认新系统上线期间产生的新增数据范围（时间段、涉及模块）。
- [ ] 评估是否需要将新系统数据反向同步回原系统。
- [ ] 如需恢复数据库，使用切换前全量备份恢复 PostgreSQL。
- [ ] 如需保留新系统数据，导出关键表（客户、合同、用电数据、询价）供后续补录。
- [ ] 校验数据一致性及业务连续性（关键记录数、金额汇总）。
- [ ] 数据恢复完成后，由业务负责人签字确认。
- [ ] 更新备份策略，避免类似问题再次发生。
